# 广告数据洞察策略设计文档

**日期**：2026-04-27  
**作者**：Claude Code  
**状态**：待审阅

## 一、背景与目标

### 项目背景
基于 `data_problems`（9类广告投放问题案例）和 `data_advantages`（7类广告数据亮点案例）两份知识库，在现有广告报表多Agent系统中增加自动化洞察能力。

### 核心目标
1. 主动发现广告数据中的问题和亮点
2. 提供数据驱动的可执行建议
3. 集成到现有查询流程中，无需用户额外操作
4. 兼顾分析速度和分析深度

## 二、总体架构

### 架构选型：混合方案（规则引擎 + LLM）

```
┌─────────────────────────────────────────────────────────┐
│                  广告报表多 Agent 系统                   │

│  │  查询 Agent  │───▶│  数据处理    │───▶│ 前端展示  │  │
│  └──────────────┘    └──────────────┘    └───────────┘  │
│                              │                          │
│                              ▼                          │

│                    │   洞察分析模块   │                 │

│                              │                          │
```

### 核心设计原则

1. **规则优先**：先用规则引擎快速识别确定性问题/亮点，零延迟、零成本
2. **LLM 增强**：对规则识别出的结果用 LLM 生成自然语言解释和建议
3. **二次扫描**：LLM 同时做开放式扫描，发现规则外的异常模式
4. **结果可追溯**：每个洞察必须附带数据证据和来源说明

## 三、规则引擎层设计

### 问题识别规则库（9类）

| 规则ID | 规则名称 | 判定条件 | 严重程度 |
|--------|----------|----------|----------|
| P01 | 受众定位偏差 | CTR正常(1%-3%) 且 CVR < 0.5% 且 跳出率 > 80% | 高 |
| P02 | 创意疲劳衰减 | 连续3天CTR下降 > 20% 且 平均频次 > 5次 | 中 |
| P03 | 时段投放浪费 | 某时段消耗占比 > 30% 且 CPA 是平均值的 2倍+ | 高 |
| P04 | 频次失控 | 高频用户(曝光>10次)消耗占比 > 30% 且 转化贡献 < 5% | 高 |
| P05 | 流量作弊嫌疑 | CTR > 10% 且 跳出率 > 90% 且 深夜流量占比 > 50% | 高 |
| P06 | 出价策略异常 | 预算1小时内烧完 或 CPC 环比上涨 > 100% | 中 |
| P07 | 地域渗透饱和 | 核心城市Reach > 80% 且 CPA 是外围城市的 1.5倍+ | 中 |
| P08 | 设备兼容问题 | 某设备/OS版本的CTR/CVR 是其他版本的 1/3 以下 | 中 |
| P09 | 竞品冲击 | CPC环比上涨 > 50% 且 CTR 下降 > 30% | 中 |

### 亮点识别规则库（10类）

| 规则ID | 规则名称 | 判定条件 | 价值等级 |
|--------|----------|----------|----------|
| A01 | CTR 表现优异 | CTR 是账户历史平均值的 3倍+ 且 7天内无明显衰减 | 高 |
| A02 | CVR 表现优异 | CVR 是行业平均值的 3倍+ 且 不随CTR波动 | 高 |
| A03 | CPC 成本优势 | CPC 低于行业均价 50%+ 且 提预算后涨幅 < 20% | 高 |
| A04 | 消耗曲线健康 | 预算利用率 > 90% 且 每小时消耗波动 < 20% | 中 |
| A05 | 频次控制良好 | 平均频次 1.5-2.5 且 频次>5次的消耗占比 < 5% | 中 |
| A06 | 转化节奏理想 | 80%+ 转化发生在 24小时内 | 中 |
| A07 | 分时段反差亮点 | 某时段CVR 是整体的 3倍+ | 高 |
| A08 | 分设备反差亮点 | 某设备ROI 是其他设备的 2倍+ | 高 |
| A09 | 分素材反差亮点 | 低CTR(<1%)但高CVR(>5%) | 高 |
| A10 | 展示份额优势 | 展示份额持续 > 60% | 中 |

### 规则引擎技术要点

- 规则阈值可配置，支持按广告主行业自定义
- 规则支持权重和优先级排序
- 每条规则匹配后返回结构化结果对象
- 不匹配时静默跳过，无额外性能开销
- 支持规则热更新，无需重启服务

## 四、LLM 洞察层设计

### LLM 层三大职责

#### 1. 规则结果的自然语言解读

**Prompt 模板**：
```
任务：将广告数据洞察转化为自然语言建议

参考知识库：
- data_problems：9类常见投放问题
- data_advantages：10类数据亮点

规则匹配：{rule_id} - {rule_name}
数据证据：{evidence_data}

输出格式：
【{问题/亮点}类型】{名称}
【数据证据】{具体数据说明}
【建议措施】{可执行的优化建议}
```

#### 2. 开放式模式扫描

**Prompt 模板**：
```
任务：扫描广告数据，发现规则未覆盖的异常模式或亮点

参考知识库：
- data_problems：9类常见问题
- data_advantages：10类数据亮点

输入数据：
{当前查询的广告数据，结构化格式}

输出要求：
1. 列出你发现的任何有趣模式、异常点、反差现象
2. 每个洞察必须附带具体数据证据
3. 标注置信度（高/中/低）
4. 不超过5条最重要的发现
```

#### 3. 洞察聚合与排序

合并规则引擎结果和 LLM 发现，按以下优先级排序：
1. 严重程度/价值等级
2. 置信度
3. 数据证据充分性

### Prompt 工程要点

- **Few-Shot 学习**：嵌入 data_problems/data_advantages 中的真实案例
- **结构化输出**：强制 JSON 格式输出，便于前端解析和渲染
- **证据要求**：每个洞察必须引用具体数值，禁止空泛结论
- **置信度标注**：LLM 发现的洞察需要标注置信度

## 五、系统集成方案

### 数据流向

```
用户查询
    ↓
查询 Agent → 获取广告数据（当前查询范围）
    ↓
数据处理层（维度解析、格式化）
    ↓
洞察分析模块入口
    ├─→ 规则引擎同步扫描（<10ms）
    └─→ LLM 洞察层异步处理
    ↓
洞察聚合（去重、排序、分级）
    ↓
前端渲染（文字报告 + 图表高亮标注）
```

### 新增组件

#### 1. Insight Agent
作为 LangGraph 中的独立节点：
- 输入：查询结果数据、查询上下文（广告主、时间、维度）
- 输出：结构化洞察列表（problems、highlights、summary）

#### 2. 后端 API

```python
POST /api/insights/analyze

Input:
{
    "query_data": {...},      // 当前查询结果
    "query_context": {        // 查询上下文
        "advertiser_id": "...",
        "time_range": {...},
        "dimensions": [...]
    },
    "options": {
        "enable_llm_scan": true,    // 是否启用LLM深度扫描
        "min_severity": "medium"    // 最低显示严重程度
    }
}

Output:
{
    "problems": [
        {
            "id": "P01",
            "type": "受众定位偏差",
            "severity": "high",
            "confidence": 0.95,
            "evidence": "...",
            "suggestion": "...",
            "source": "rule_engine | llm"
        }
    ],
    "highlights": [...],
    "summary": "共发现 3 个问题，2 个亮点...",
    "llm_insights": [...]    // LLM额外发现
}
```

#### 3. 前端扩展

- 增加"智能洞察"折叠卡片，位于查询结果上方
- 问题：红色🔴（高）、黄色🟡（中/低）
- 亮点：绿色🟢
- 点击展开：数据证据 + 优化建议
- 支持筛选：仅显示高优先级、仅显示问题、仅显示亮点

### 性能优化

- 规则引擎同步执行（<10ms）
- LLM 分析异步执行，前端显示"正在生成深度分析..."
- 缓存相同查询参数的洞察结果（TTL: 1小时）
- 支持按查询复杂度自动降级（大数据量时跳过LLM深度扫描）

## 六、测试策略

### 单元测试
- 每条规则的边界条件测试（如：CTR 刚好下降 20% 是否触发）
- 规则引擎性能基准测试

### 集成测试
测试用例来源于知识库：
- 输入 data_problems 中的案例数据，验证能否识别对应问题
- 输入 data_advantages 中的案例数据，验证能否识别对应亮点
- 准确率目标：> 90%

### LLM 输出质量评估
评估维度：
1. 数据准确性（证据是否正确）
2. 建议可操作性（是否具体可执行）
3. 语言自然度（是否符合分析师口吻）

## 七、演进路线图

| 阶段 | 功能范围 | 时间估计 | 交付标准 |
|------|----------|----------|----------|
| MVP | 规则引擎（核心5条规则）+ 基础LLM解读 | 1-2周 | 能识别主要问题和亮点 |
| v1.0 | 完整 9+10 规则 + LLM二次扫描 + 前端展示 | 2-3周 | 覆盖知识库所有场景 |
| v1.5 | 洞察置信度评估 + 用户反馈闭环 | 1-2周 | 支持用户标注洞察质量，持续优化 |
| v2.0 | 自动归因下钻（发现问题后自动查细分维度找原因） | 2-3周 | 根因分析能力 |

## 八、边界与限制

### 当前范围
- 仅分析当前查询结果数据（不做历史对比、行业对比）
- 主动触发（每次查询自动运行）
- 输出自然语言报告 + 前端可视化

### 暂不包含
- 自动执行优化建议（仅给出建议）
- 跨广告主对比分析
- 预测性洞察（预判未来走势）
