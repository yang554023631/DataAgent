**日期：** 2026-05-02
**版本：** v1.0
**状态：** 设计中

## 一、项目背景与目标

在现有的广告报表多 Agent 系统基础上，新增业务知识 RAG（检索增强生成）功能，使 Agent 能够：
1. 智能识别用户意图是「报表查询」还是「知识问答」
2. 对知识类问题，通过 RAG 检索广告业务知识库并给出专业回答

### 核心目标
- 支持约 50-100 篇业务知识文档的检索
- 检索准确率达到 90% 以上
- 回答质量达到业务专家水平

## 二、整体架构设计

```
用户输入
    ↓
┌─────────────────────────────────────────┐
│         🧭 意图路由 Agent               │
│  判断：报表查询 OR 知识问答              │
└─────────────────────────────────────────┘
        ↓                      ↓
┌───────────────┐      ┌───────────────────────┐
│ 报表 Agent 流 │      │   📚 RAG Agent 流     │
│ （现有逻辑）  │      │   - 文档检索          │
│               │      │   - 重排序           │
│               │      │   - 答案生成         │
└───────────────┘      └───────────────────────┘
        ↓                      ↓
        └───────────┬──────────┘
                    ↓
              结果展示给用户
```

### 核心设计决策
| 决策项 | 选型 | 说明 |
|--------|------|------|
| 向量数据库 | PostgreSQL + pgvector | 本地编译安装到 `~/Tools/pgsql/` |
| 嵌入模型 | 商用 Embedding API | OpenAI text-embedding-3-small 或国内同类 |
| 重排序模型 | bge-reranker-base | 本地运行，提升检索效果 |
| 检索策略 | 向量检索 + Reranker 重排序 | Top 20 召回 → Top 3-5 精排 |
| 意图模式 | 互斥模式 | 一个问题要么走报表，要么走 RAG |
| 文档管理 | 目录文件自动同步 | `docs/rag/` 目录下 Markdown 文件 |
| 同步机制 | 脚本支持全量/增量同步 + 目录监听 | 变化自动更新向量库 |

## 三、数据库与向量存储设计

### 3.1 部署配置
- **安装路径：** `~/Tools/pgsql/`（独立目录，不影响系统 PostgreSQL）
- **数据目录：** `~/Tools/pgsql/data/`
- **端口：** 默认 5432（可配置）
- **版本：** PostgreSQL 16 + pgvector 0.7.0

### 3.2 数据库表设计

```sql
-- 文档表
CREATE TABLE rag_documents (
    id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    title VARCHAR(500) NOT NULL,           -- 文档标题
    file_path VARCHAR(1000) NOT NULL,      -- 源文件路径
    doc_type VARCHAR(100),                 -- 文档类型：business/operation/system
    version VARCHAR(50),                   -- 版本号
    is_active BOOLEAN DEFAULT true,        -- 是否启用
    created_at TIMESTAMP DEFAULT NOW(),
    updated_at TIMESTAMP DEFAULT NOW(),
    last_synced_at TIMESTAMP,              -- 最后同步时间
    UNIQUE(file_path)
);

-- 文档分片表
CREATE TABLE rag_chunks (
    id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    doc_id UUID REFERENCES rag_documents(id) ON DELETE CASCADE,
    chunk_index INT NOT NULL,              -- 分片序号
    content TEXT NOT NULL,                 -- 分片内容
    content_hash VARCHAR(64) NOT NULL,     -- 内容 hash（用于增量更新）
    embedding vector(1536),                -- 向量，维度根据模型调整
    created_at TIMESTAMP DEFAULT NOW(),
    UNIQUE(doc_id, chunk_index)
);

-- 向量索引（HNSW 索引，适合高维向量快速检索）
CREATE INDEX ON rag_chunks USING hnsw (embedding vector_cosine_ops);
```

### 3.3 同步策略
- **全量同步：** 扫描目录 → 解析 → 分片 → 生成向量 → 写入
- **增量同步：** 对比文件最后修改时间 + content_hash，只更新变化的文档
- **自动同步：** inotify 监听目录变化，自动触发增量同步

## 四、RAG 核心模块设计

### 4.1 文档处理流水线
```
Markdown 文件
    ↓
  解析器（识别标题、代码块）
    ↓
  分片策略（按标题层级分片，智能重叠 ~20%）
    ↓
  Embedding API 生成向量
    ↓
  写入 pgvector
```

### 4.2 检索流水线
```
用户问题
    ↓
  问题重写（可选，用于多轮对话）
    ↓
  生成问题向量
    ↓
  向量检索 Top 20
    ↓
  Reranker 重排序（bge-reranker-base）
    ↓
  选出 Top 3-5 相关分片
    ↓
  构造 Prompt 给 LLM 生成回答
```

### 4.3 核心模块划分
| 模块 | 文件 | 职责 |
|------|------|------|
| 向量生成 | `rag/embedding.py` | 调用 Embedding API，支持多模型切换 |
| 检索引擎 | `rag/retriever.py` | 向量检索 + Reranker 重排序逻辑 |
| 文档分片 | `rag/splitter.py` | Markdown 智能分片，按标题层级 |
| 同步脚本 | `rag/sync.py` | 全量/增量/监听模式同步 |
| Agent 节点 | `rag/agents.py` | LangGraph RAG 相关节点定义 |

## 五、LangGraph 集成设计

### 5.1 State 扩展
```python
class AdReportState(TypedDict):
    # ... 现有字段 ...
    # 新增 RAG 相关字段
    query_type: Literal["report", "knowledge"]  # 查询类型：报表/知识
    rag_query: str                               # 重写后的 RAG 查询
    rag_context: list[str]                       # 检索到的相关文档
    rag_answer: str                              # RAG 生成的答案
```

### 5.2 图节点扩展
```
                    ┌───────────────┐
                    │  用户输入     │
                    └───────┬───────┘
                            ↓
                    ┌───────────────┐
                    │  意图路由     │  ← 新增节点
                    │  (Router)     │
                    └───────┬───────┘
                ┌───────────┴───────────┐
                ↓                       ↓
        ┌───────────────┐       ┌───────────────┐
        │  报表查询流   │       │  RAG 检索     │  ← 新增节点
        │ （现有逻辑）  │       └───────┬───────┘
        └───────┬───────┘               ↓
                │               ┌───────────────┐
                │               │  RAG 回答生成 │  ← 新增节点
                │               └───────┬───────┘
                └───────────┬───────────┘
                            ↓
                    ┌───────────────┐
                    │  结果返回     │
                    └───────────────┘
```

### 5.3 意图路由 Prompt
```
你是一个智能路由助手，根据用户问题判断应该走哪个分支：

- 如果用户想看数据、报表、图表、趋势、对比、分析某个指标、查某个维度 → 走 "report"
- 如果用户问概念、术语、使用方法、帮助、说明、定义、解释是什么 → 走 "knowledge"

当前用户问题：{user_input}
请只返回：report 或 knowledge
```

## 七、错误处理与测试策略

### 7.1 错误处理
| 场景 | 处理策略 |
|------|---------|
| 向量 API 调用失败 | 重试 3 次 + 指数退避，失败返回友好提示 |
| 数据库连接失败 | 健康检查 + 自动重连 |
| 无相关文档 | 返回"抱歉，暂时没有找到相关帮助文档，您可以试试其他问题" |
| 意图识别不确定 | 第一版暂不处理，后续可加澄清节点 |

### 7.2 测试策略
- **单元测试**：分片算法、向量生成、检索逻辑
- **集成测试**：端到端同步 + 检索流程
- **效果评估测试**：测试用例集评估检索准确率
  ```
  Q: "什么是 CTR？" → 应该命中 CTR 定义文档
  Q: "怎么看渠道对比？" → 应该命中渠道对比帮助文档
  ```
