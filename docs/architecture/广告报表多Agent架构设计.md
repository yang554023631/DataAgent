# 广告报表多 Agent 架构设计与功能规划

## 一、现有系统基础分析

### 已实现功能
- **后端**：FastAPI + Elasticsearch，`/api/report/query` 接受 `QueryRequest` → 构造 ES DSL → 执行查询 → 计算派生指标 → 返回结果
- **前端**：React + ECharts，解析 URL 参数 → 渲染条件编辑区 + 图表 + 表格
- **两种模式**：static（只读）和 interactive（可编辑）

### 可复用资产
| 资产 | 用途 |
|------|------|
| `execute_query(QueryRequest)` | 执行报表查询 |
| `build_query(QueryRequest)` | 构造 ES DSL |
| `get_all_advertisers()` | 获取广告主列表 |
| 完整类型定义 | `Filter/GroupBy/Metric/ChartConfig` |

### ES 索引结构
- 事实表：`ad_stat_data`（通用）、`ad_stat_audience`（受众细分）
- 维度表：`advertiser`, `campaign`, `adgroup`, `creative`

## 二、LangGraph 多 Agent 架构设计

### 整体数据流
```
用户自然语言输入
    ↓
┌─────────────────────────────────────────┐
│         🤖 意图理解 Agent               │
│  任务：把自然语言 → 结构化查询意图       │
│  输出：{时间范围, 维度集合, 指标集合}   │
└─────────────────────────────────────────┘
    ↓ 有歧义？
┌─────────────────────────────────────────┐
│         ❓ 澄清交互节点                 │
│  向用户确认："你要的效果是看 CTR 还是   │
│  CVR？分渠道看吗？"                     │
└─────────────────────────────────────────┘
    ↓ 澄清后
┌─────────────────────────────────────────┐
│         📋 查询规划 Agent               │
│  任务：生成合法的 QueryRequest 对象     │
└─────────────────────────────────────────┘
    ↓
┌─────────────────────────────────────────┐
│         ✅ 校验执行节点                 │
│  调用 execute_query() 查数据           │
└─────────────────────────────────────────┘
    ↓
┌─────────────────────────────────────────┐
│         📊 分析报告 Agent               │
│  - 异常检测、环比同比、生成结论         │
└─────────────────────────────────────────┘
    ↓
┌─────────────────────────────────────────┐
│         🎯 结果展示 Agent               │
│  - 生成报表 URL、展示摘要              │
│  - 接收用户追问 → 回到查询规划          │
└─────────────────────────────────────────┘
```

### LangGraph State 设计
```python
class AdReportState(TypedDict):
    user_input: str                    # 用户原始输入
    intent: dict                       # 解析后的查询意图
    clarification_needed: bool         # 是否需要澄清
    clarification_question: str        # 澄清问题
    user_clarification: str            # 用户澄清
    query_request: QueryRequest        # 最终查询计划
    query_result: dict                 # 执行结果
    analysis: str                      # 分析结论
    report_url: str                    # 生成的报表 URL
```

### Tool 设计
```python
@tool
def query_ad_report(query_request: dict) -> dict:
    """执行广告报表查询，返回查询结果"""
    req = QueryRequest(**query_request)
    return execute_query(req)

@tool
def get_advertiser_list() -> list:
    """获取所有广告主列表"""
    return get_all_advertisers()
```

## 三、功能扩展规划（按优先级）

## 🔴 第一优先级：智能分析类（最能体现人机交互价值）

### 1. 自动异常检测与根因下钻
**是什么：**
查询完数据后，系统主动告诉你："发现 A 渠道上周 CTR 环比下降 32%，已自动下钻分析：主要是创意 ID 12345 的 CTR 从 5% 掉到 1%"

**为什么重要：**
- 传统报表需要用户自己发现问题，这个功能把"人找问题"变成"问题找人"
- 90% 的用户看报表第一件事就是找"哪里不对"，这一步可以帮用户省 80% 的时间

**实现思路：**
- Data Analyst Agent 拿到数据后运行异常检测算法（环比/同比、Z-score）
- 发现异常 → 自动发起下钻查询（调用已有的 execute_query）
- 从高维度 → 低维度逐层拆解（渠道 → 广告组 → 创意）
- 生成根因分析结论

---

### 2. 智能归因分析
**是什么：**
用户问："为什么上周花费涨了 50%？"
系统答："花费上涨主要来自三个方面：60% 来自新增的 A 渠道，30% 来自 B 渠道 CPC 从 3 块涨到 5 块，10% 来自其他渠道流量增加"

**为什么重要：**
- 用户不需要自己一层层拆解数据，系统自动做维度贡献度分析
- 这是高级分析人员才有的能力，现在普惠给所有用户

**实现思路：**
- Query Planner 自动生成多维度的分组查询（按渠道、按广告组、按创意）
- 归因 Agent 计算各维度的贡献度（delta 拆分）
- 按贡献大小排序输出结论

---

### 3. 对比分析助手
**是什么：**
用户："对比一下 A 渠道和 B 渠道上月的表现"
系统：自动生成对比表格 + 差异百分比 + 雷达图 + 文字总结："A 渠道 CTR 高 15%，但 CVR 低 8%，整体 ROI B 渠道更好"

**为什么重要：**
- 对比是最常见的分析动作，但传统报表需要用户手动查两次然后自己算差异

## 🟠 第二优先级：交互体验增强（让对话更像人）

### 4. 上下文理解与增量修改
**是什么：**
```
用户：看上周移动端的 CTR
系统：[展示按天的 CTR 折线图]

用户：分渠道看  ← 不需要重复说"上周"、"移动端"、"CTR"
系统：[自动加上渠道维度，展示渠道对比柱状图]

用户：把最差的那个渠道再按创意拆解  ← 系统知道"最差"指的是 CTR 最低
系统：[自动过滤该渠道 + 按创意分组排序]
```

**为什么重要：**
- 人说话就是带上下文的，不需要每次都说全
- 这是"对话式"和"命令式"最本质的区别

**实现思路：**
- LangGraph State 持久化当前查询上下文（时间范围、过滤条件、当前维度、指标）
- 意图理解 Agent 不是每次生成新的 QueryRequest，而是做增量修改
- 维护"当前聚焦对象"的概念（比如刚才说的"最差的那个渠道"）

---

### 5. 业务术语理解与模糊匹配
**是什么：**
| 用户说的话 | 系统理解成 |
|-----------|-----------|
| "投产比" | ROI = 转化价值 / 花费 |
| "点击率" | CTR |
| "安卓端" | audience_os=2 (Android) |
| "大渠道" | 花费 Top3 的渠道 |
| "效果好" | CTR > 3% 且 CVR > 5% |

**为什么重要：**
- 用户说自己的话，不是系统的术语
- 减少 90% 的澄清问题

---

### 6. 查询优化建议
**是什么：**
用户："看过去一年每天的明细"
系统："这个查询会返回 365×维度数 条数据，预计需要 8 秒，要不要改成按月汇总？"

**其他场景：**
- "你选了 5 个分组维度，图表可能会很挤，要不要减少到 3 个？"
- "这个时间范围没有数据，要不要扩大到最近 30 天？"

## 🟡 第三优先级：报表生产力（让做报表更快）

### 7. 报表模板库与一键生成
**是什么：**
用户："生成一份上周的运营日报"
系统自动生成完整报告，包含：
1. 核心指标总览（曝光/点击/花费/CTR/CVR）
2. 渠道效果排名（Top5 / Bottom5）
3. 创意表现排行
4. 异常点说明与建议

**内置模板：**
- 日报/周报/月报（自动算环比同比）
- 渠道分析报告
- 受众画像报告
- 创意效果分析报告

---

### 8. 自然语言图表定制
**是什么：**
```
用户：把这个折线图改成柱状图
用户：Y 轴只留 CTR 和 CVR
用户：按 CTR 降序排，只看 Top10
用户：把这两个系列的颜色换一下
```

**为什么重要：**
- 不需要用户去 UI 上点 5 次菜单，直接说就行

---

### 9. 智能维度推荐
**是什么：**
系统："你现在看的是按渠道的 CTR，要不要试试按受众性别拆分？数据显示不同性别差异达到 40%"
系统："广告组维度还有更细的信息，要继续下钻吗？"

**实现思路：**
- 分析 Agent 评估各维度的信息增益（方差、区分度）
- 推荐能带来最大新信息的维度组合

## 📊 落地优先级建议

**第一阶段（1-2 周）：先做最容易出效果的**
1. ✅ 上下文理解与增量修改（改 State 逻辑就行）
2. ✅ 自动异常检测（已有数据，加个算法就行）
3. ✅ 业务术语映射（加个词典就行）

**第二阶段（3-4 周）：核心分析能力**
4. 🔄 根因下钻分析
5. 🔄 对比分析助手
6. 🔄 报表模板库

**第三阶段（长期）：高级功能**
7. 归因分析
8. 预测与告警
9. 创意诊断

## 四、架构优势总结

| 角度 | 为什么多 Agent + LangGraph 好 |
|------|-----------------------------|
| **复用性** | 100% 复用已有的后端查询逻辑，不需要重写 ES 查询 |
| **可观测性** | 每个节点的输出都可以独立检查，方便调试 |
| **可迭代性** | 可以先做简单版本，再逐步增强每个 Agent 的能力 |
| **性能** | LLM 只做规划，真正的查询还是由优化过的 Python 代码执行 |
| **用户体验** | 支持自然语言 + 上下文理解 + 人机闭环 |
